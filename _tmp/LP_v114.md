02 The Problem

Financial standing does not travel onchain. A trader can build years of disciplined history and still arrive at every new protocol as an unknown wallet.

Autonomous finance has the same problem: agents can transact, but their financial standing still resets across applications.

Not knowing an operator's track record blocks delegated agent authority, portable leverage and collateral treatment, undercollateralized credit, and cross-application financial identity.

Previous approaches failed in three ways. Reputation scores were delivered through oracles, so applications could read or ignore them at will. Feedback-based agent reputation was cheap to farm and said nothing about whether real capital was ever handled well. Guardrail systems enforced limits an operator configures, not Network Standing an operator earns.

Nasun's answer is runtime-guaranteed Network Standing. Human and agent activity contribute to one persistent financial record. That record is canonical runtime state, inherited by every native application as a floor every application must honor. An operator's Network Standing follows them across the environment. An application cannot deny it. An operator cannot grant themselves more than they have earned. Financial history compounds across applications, across agents, and across time, and it is earned.

03 Why 2026

Three shifts unlock Nasun now.

Agent identity is solved, agent trust is not. ERC-8004 went live on Ethereum mainnet in January 2026 with portable identity and a feedback-based reputation registry. AgentKit and World address who is behind an agent and how it pays. None produce a portable, enforceable answer to how much Network Standing an operator has earned, and feedback is farmable. The missing layer is Network Standing derived from real economic behavior and guaranteed where execution happens.

Infrastructure is commoditized. Production-ready consensus, onchain orderbooks, zkLogin, and passkeys make launching an execution environment a matter of weeks. The hard problems left are behavioral and economic: whether humans and the agents they deploy can build portable Network Standing, and whether that standing can be guaranteed at execution rather than treated as optional application logic.

Autonomous capital is already moving. Agents trade, manage positions, and execute strategies onchain now. Every delegation is a risk decision with nothing tying what the software may do back to the operator's own track record. That is the gap Nasun fills.

04 How Nasun Works

Nasun runs on its own Move-based runtime where earned Network Standing is guaranteed at the runtime level rather than left to application logic. A participant can act directly or deploy agents. Both feed the same persistent record, and that record establishes what the participant and their agents have earned the right to access across the environment.

Every economic action produces behavioral scores. Scores attach to specific agents and behaviors and roll into a unified participant standing, so an application can read an agent's own track record directly, not only its operator's. That standing resolves into a Network Standing tier. Native applications inherit a shared nasun::tier module that exposes operator-derived Standing floors: fee tier, agent capital cap, execution permissions, venue access. Applications query the runtime function and the runtime verifies at execution that floors were honored. Applications may extend permissions above the floor at their own discretion. They cannot deny the floor.

Standing floors apply per agent, not only per operator. Each agent the operator deploys builds its own execution history through its AERs, and that history contributes to the agent's tier independent of the operator's. A high-tier operator does not automatically grant a new agent high-tier Standing; the agent earns its own standing through its own behavior. An agent's tier is a function of both records. 

External applications, outside the Nasun runtime, may consume Nasun history and AER data as input to their own independent decisions. They are not subject to runtime enforcement.

Executor operators use dual-axis authority: tier \= min(stake\_tier, reputation\_tier). Stake reflects NSN committed. Reputation rises with verified execution, falls on error or timeout, and decays after inactivity. Capital alone cannot buy top tier, and behavior alone cannot reach it either. This is live in executor\_tier.move today.

Every agent execution emits an AIExecutionReport (AER) to the runtime, signed and indexed onchain, inspectable against the budget, operator, model class, venue, and outcome it records. Reputation, participant history, and runtime billing settle from the same primitive. Replay protection and per-agent and global kill-switches are deployed.

V1 scoring uses simple inputs (account age, verification signals, lifetime spend); V2 expands into realized PnL, trading volume, prediction accuracy, and imported cross-chain history. Scoring upgrades are forward-only and do not reset accumulated standing. Nasun already indexes verified activity across Ethereum, Solana, and Sui, so participants import existing behavioral history instead of starting from zero.

07 The Execution Environment

Nasun runs on its own Move-based runtime, forked from Sui and built on Mysticeti consensus, for two reasons: portable guarantees and execution depth.

The first is portable guarantees. The product breaks if applications can opt out of honoring Network Standing. A reputation score delivered through an oracle is optional: one application honors it, another ignores it. That is why oracle-based reputation has not become real financial infrastructure. A runtime that only enforces limits an operator configures per agent is enforcing settings, not standing. Nasun enforces what a configurable guardrail cannot: a portable floor of Network Standing earned from a persistent track record. The rules governing what each tier guarantees live in the runtime, and every native application inherits the same earned-history floor by default. Applications compete above the floor on product quality, liquidation engine, fees, UX, and how much additional authority they choose to extend.

Enforcement lives at the runtime policy layer, not the consensus layer. Native applications inherit the shared nasun::tier module that exposes operator-derived Network Standing floors. The runtime verifies at transaction execution that floors were honored. Consensus is unchanged; what changes is the canonical policy layer above it.

Owning the runtime is what makes tier the economic substrate of the protocol rather than a feature within it. NSN emissions weighted by tier, gas sponsorship gated by tier, validator and executor staking aligned with the behavioral economy's growth, and cross-application canonical state are protocol-level interlocks. Implemented as an application on a shared L1, the tier mechanism would still work mechanically through Move composition, but it would not capture protocol-level value because the surrounding token economics, gas economics, and consensus-adjacent state would belong to the host chain. The Hyperliquid precedent demonstrates the pattern: when financial execution and protocol economics need to align, the runtime form is what enables that alignment.

General-purpose chains will not occupy this lane. Their value proposition is neutrality. A base chain that guaranteed one canonical Network-Standing standard across its applications would be competing with its own ecosystem. The lane is open because of a business-model conflict, not an information gap.

The second reason is execution depth. Cross-margin, conditional orders, agent-triggered execution, reward routing, and capital floors all run at the runtime layer rather than being stitched together through external keepers. Richer execution produces richer behavioral history, and richer history is what makes the Network-Standing layer valuable.

08 Network Standing Roadmap

What an operator may access is a floor the runtime guarantees, scaled to earned history, not a permission an operator configures or buys. Stake gates participation tier. History earns Network Standing.

V1, Participation Layer. Live on devnet today, shippable as a mainnet business within four to six months. Verified history and stake depth establish runtime-guaranteed floors on fee tiers, emissions weighting, staking boosts, agent capital floors, and execution permissions. These floors are queryable as canonical primitives by any native application and verified at transaction execution. New users retain a competitive baseline at tier 0; tier benefits stack above the baseline. This stage does not require any application to take credit risk and does not depend on any later stage.

V2, Capital Coordination. Plausible within the post-testnet window. Inside Nasun-native finance, the Network-Standing floor extends to leverage minimums, liquidation parameter floors, cross-margin availability, agent capital floors, and routing-priority floors. The runtime guarantees parameters and availability; applications retain ownership of their margin engines, liquidation logic, and underwriting decisions. V2 distinguishes mechanical floors (parameters the runtime can verify) from data-portability primitives (credit, pricing, spread), with only the former enforced by the runtime and the latter exposed as queryable history for applications to consume in their own decisions.

V3, Financial Trust Infrastructure. The long-term direction, trajectory not current capability. Behavioral history becomes an input external venues consume to price operator risk: leverage ceilings, RFQ access, cross-application credit decisions those venues make themselves. Here Nasun is a behavioral data layer consumed by lenders and venues, not a lender itself and not an adjudicator of access.

Devnet behavioral history carries into mainnet tier standing through identity-bound import. The specific carry-over policy is published ahead of mainnet. Scoring algorithm changes (V1 to V2) are forward-only runtime updates and do not reset accumulated standing.

09 The App Directory

The App Directory is a curated set of applications a user activates for deeper engagement. Activation creates recurring financial activity (trading, game participation). Sustained activity holds reward weighting at maximum; neglect lets it decay. On devnet this accrues as points; on mainnet it becomes emissions boosts stacking above the baseline staking rate. Engagement is optional: a user who only stakes still earns baseline emissions.

This is distinct from the test-contributor points stream noted in Section 05, which rewards bug reports and feedback. That stream is how a devnet test program is operated.

Native applications built inside Nasun, Pado and GoStop today, run in the runtime and inherit the operator-derived Standing floor mechanically. They cannot deny what the operator has earned. Above the floor, they set their own policies, risk parameters, and product caps. Competition above the floor on UX, liquidation engine, fees, community, and additional permissions is the ecosystem dynamic the runtime is designed to enable. The floor's value is portability of earned status across venues, not uniformity of treatment.

External applications enter in two phases. Phase one lists established applications from supported chains, such as Jupiter or Aave, for user capture. Their users are credited for history the wallet module already ingests. These applications do not honor Nasun-runtime floors and Nasun does not claim otherwise. Phase two begins once the userbase is large enough to be leverage. Newer applications need distribution and a way to price users by track record. Nasun supplies both. The external app remains the decision-maker, consuming Nasun history as input to its own decisions on its own platform.

This is sequencing, not a live system. Native floor enforcement is live at devnet maturity. The external tier is the planned model for testnet and mainnet, gated on userbase scale Nasun does not yet have.

10 Competition and Moat

Cred Protocol and Spectral built onchain credit scores, but oracle-based reputation remains optional. Nasun guarantees instead, embedding portable Network Standing into the runtime as floors every native application must honor.

A separate class of projects, including AgentKit spend permissions and Skyfire-style payment rails, enforces spending limits inside their own runtime. The distinction matters: those systems enforce limits an operator configures per agent, not portable Network Standing earned from a persistent record. Nasun does not enforce settings. It guarantees earned standing.

3Jane and Wildcat built undercollateralized lending products with their own underwriting. Nasun does not compete with them; it is building the data layer systems like those would consume. World and Coinbase AgentKit solve agent identity and payments: who an agent belongs to and how it transacts. They do not answer how much Network Standing the operator behind that agent has earned, portable across venues.

The moat is not the runtime itself; execution environments are commoditized. The differentiator is a portable Network-Standing floor where behavioral history compounds across applications and participants over time. Each new integration adds behavioral data that makes the floor more valuable to operators and more attractive to applications competing for high-tier users above it. The moat does not depend on external chains adopting Nasun's standard. It requires enough economic activity inside Nasun's own environment for portable Network Standing to become valuable infrastructure.

The narrower claim, and the foundation the moat builds on, is that the system has been built, integrated, and operated end-to-end, and that the operational substrate the moat depends on already exists. The mechanism is real today. The scale that makes it defensible is what the roadmap is for.
