# Lesson Learned

실제 코드 변경에서 구체적이고 근거 있는 소프트웨어 엔지니어링 교훈을 추출합니다.
강의가 아니라 거울 -- 코드가 이미 보여주는 것을 반영합니다.

## Phase 1: 스코프 결정

| 스코프 | Git 명령 | 사용 시점 |
| ------ | -------- | --------- |
| Feature 브랜치 | `git log main..HEAD --oneline` + `git diff main...HEAD` | main이 아닌 브랜치에서 |
| 최근 N개 커밋 | `git log --oneline -N` + `git diff HEAD~N..HEAD` | main에서 (기본 N=5) |
| 특정 커밋 | `git show <sha>` | 사용자가 특정 커밋 지정 |
| 작업 중 변경 | `git diff` + `git diff --cached` | 커밋 전 |

**기본 동작**:
- Feature 브랜치 → 브랜치 커밋 vs main 분석
- main 브랜치 → 최근 5개 커밋 분석

## Phase 2: 변경사항 수집

1. `git log`로 커밋 목록과 메시지 확인
2. `git diff`로 전체 diff 확인
3. diff가 500줄 이상이면 `git diff --stat`으로 요약 후 상위 3-5개 파일만 상세 확인
4. **커밋 메시지를 주의 깊게 읽기** -- raw diff가 놓치는 의도가 담겨 있음
5. 변경된 파일만 읽기. 전체 레포를 읽지 않음

## Phase 3: 분석

**지배적 패턴** 하나를 식별 -- 이 변경에서 가장 교훈적인 것.

찾아볼 것:
- **구조적 결정** -- 코드가 어떻게 조직되었는가? 왜 그 경계인가?
- **트레이드오프** -- 무엇을 얻고 무엇을 포기했는가?
- **해결된 문제** -- before/after의 차이, "after"가 더 나은 이유
- **개선 기회** -- "다음에는 이렇게 고려해보세요" (부드럽게)

## Phase 4: 교훈 제시

### 메인 교훈 (1개, 필수)

```markdown
## Lesson: [원칙 이름]

**코드에서 일어난 일:**
[2-3문장, 구체적 파일과 커밋 참조]

**작동하는 원칙:**
[1-2문장, SE 원칙 설명]

**왜 중요한가:**
[1-2문장, 이것 없이 무엇이 잘못될 수 있는지]

**다음을 위한 핵심:**
[한 문장, 구체적이고 실행 가능한 조언]
```

### 추가 교훈 (최대 2개, 선택)

```markdown
---

### Also: [원칙 이름]

**코드에서:** [1문장]
**원칙:** [1문장]
**핵심:** [1문장]
```

## SE 원칙 레퍼런스

| 원칙 | 설명 |
| ---- | ---- |
| SRP | 하나의 모듈은 하나의 변경 이유만 가짐 |
| DRY | 지식의 중복을 피함 (코드 중복 != DRY 위반) |
| YAGNI | 지금 필요하지 않은 것은 만들지 않음 |
| KISS | 가능한 한 단순하게, 그러나 더 단순하게는 아님 |
| Separation of Concerns | 관심사를 분리하여 독립적으로 변경 가능하게 |
| Fail Fast | 오류를 가능한 빨리 감지하고 보고 |
| Least Surprise | 코드가 예상대로 동작해야 함 |
| Composition over Inheritance | 상속보다 조합을 선호 |
| Explicit over Implicit | 암묵적인 것보다 명시적인 것을 선호 |
| Defense in Depth | 여러 계층에서 안전장치 |
| Premature Optimization | 측정 없는 최적화는 모든 악의 근원 |
| Boy Scout Rule | 코드를 발견했을 때보다 깨끗하게 남겨둠 |

## 하지 말 것

| 금지 | 이유 | 대신 |
| ---- | ---- | ---- |
| 모든 원칙 나열 | 압도적이고 일반적 | 가장 관련 있는 1-2개만 |
| 변경되지 않은 파일 분석 | 범위 초과 | diff에 집중 |
| 커밋 메시지 무시 | 의도 정보 손실 | 주요 컨텍스트로 활용 |
| 코드와 연결 안 되는 조언 | 실행 불가 | 항상 구체적 파일/라인 참조 |
| 부정적 피드백만 | 사기 저하 | 좋은 패턴 먼저, 개선 제안은 그 후 |
| 3개 초과 교훈 | 통찰 희석 | 잘 근거된 1개가 애매한 7개보다 나음 |

## 대화 스타일

- **반영적, 처방적이지 않음**: 사용자의 코드를 1차 증거로 사용
- **"~했어야 합니다" 금지**: "여기서의 접근은 ~를 보여줍니다" 또는 "다음에 이런 상황에서는 ~를 고려해보세요"
- **좋은 코드라면 칭찬**: 좋은 패턴 인식은 강화 효과
- **사소한 변경이면 솔직하게**: "이 변경은 직관적입니다 -- 깊은 교훈보다는 좋은 하우스키핑입니다."
