# Pado Score 리더보드 및 Nasun 포인트 시스템 연동 구현 계획서

**상태**: 기획 완료 (Implementation Ready)
**최근 업데이트**: 2026-04-17

---

## 1. 개요
본 문서는 기존 Pado Score 리더보드를 주간 리셋 모델로 전환하고, 성과 중심의 점수 체계로 개편하며, Nasun Ecosystem Points와 자동 연동하기 위한 상세 기술 구현 계획을 기술합니다.

## 2. 핵심 변경 사항

### 2.1 점수 체계 개편 (성과 중심)
- **성과 점수(PnL) 가중치 대폭 상향**:
    - `PER_1K_PNL`: 20 → 100 pts
    - `PER_10PCT_RETURN`: 15 → 50 pts
- **손실 방어 기제 도입**:
    - 수익률(PnL %) 기준 5% 초과 시 구간별 보너스 지급.
    - 손실률(PnL %)이 -20% 이하일 경우 성과 점수에서 20점 감점 (조작 방지 로직 적용).
- **워시 트레이딩 방지**: 동일 Identity에 귀속된 지갑 간 매칭 거래는 모든 점수 계산에서 제외.

### 2.2 주간 리셋 및 보상 시스템
- **주기**: 매주 월요일 00:00 UTC 리셋.
- **필터링 방식**: `trade_fills` 데이터를 삭제하지 않고, Aggregator가 `current_week_start`를 기준으로 필터링하여 계산.
- **보상 구조 (총 3,135 pts/주)**:
    - 1위(30), 2위(20), 3위(15), 4~50위(10), 51~100위(8), 101~200위(7), 201~300위(6), 301~400위(5), 401~500위(4).
    - **Genesis Pass 보유자**: 최종 포인트의 2배(x2) 지급.
    - **Identity 미연결 지갑**: 보상 대상에서 제외 (연결 강제 정책).

---

## 3. 단계별 구현 로드맵

### Phase 1: 데이터 모델링 및 필터링 로직 (1~2일)
- **주간 필터링 구현**: `aggregator.ts` 내 데이터 집계 시 `timestamp_ms >= current_week_start` 조건 추가.
- **테이블 구조 수정**: `trader_points`를 주차별로 식별할 수 있도록 `week_id`(예: `2026-W16`) 컬럼 추가 및 인덱스 재구성.

### Phase 2: 점수 공식 재설계 및 워시 트레이딩 필터 (1~2일)
- **가중치 상수 조정**: `leaderboard-types.ts`의 `POINTS` 상수 업데이트.
- **필터링 로직 구현**: `aggregator.ts`에서 지갑 주소 간 매칭 여부를 `identity_to_wallet_map`으로 검증하여 제외.

### Phase 3: Identity 연동 및 보상 지급 정책 구현 (2~3일)
- **매핑 검증**: `handleScoreLeaderboard` API에서 주소와 `identity_id`의 연동 여부 확인 로직 구현.
- **미연결 주소 정책**: UI에 보상 제외 안내 문구 노출.

### Phase 4: 자동 지급 파이프라인 구현 (2~3일)
- **스냅샷 크론잡**: 일요일 23:59 UTC에 주간 순위 확정 후 `weekly_score_snapshots` 테이블에 저장.
- **지급 파이프라인**: 스냅샷 데이터를 기반으로 Ecosystem Points Ledger API 연동.

---

## 4. 논리적 제약사항
1. **Identity 연동**: Pado 앱에서 거래한 지갑이 Nasun에 등록되지 않은 경우 보상을 지급하지 않으며, 이는 명시적으로 사용자에게 안내함.
2. **조작 방지**: 성과 점수 감점 로직은 조작 의심 패턴(지갑 간 거래) 차단과 병행되어야 함.
3. **인플레이션 제어**: 주간 총 지급량을 3,135 pts로 고정하여 시스템의 지속 가능성 유지.
