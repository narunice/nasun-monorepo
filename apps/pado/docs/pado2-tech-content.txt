========== WHY A NEW CHAIN ==========

[Title]
Why a New Chain

[Intro]
Pado runs on Nasun, a high-performance blockchain forked from Sui. The choice
to build a dedicated L1 is architectural, not ideological.

[Feature 1 - Object-Based State Model]
Every asset, position, and market exists as an independent on-chain object with
its own ownership and access rules. Objects can be composed, transferred, and
modified without touching global state.

[Feature 2 - Parallel Execution]
Transactions that touch independent objects execute simultaneously. A spot trade
and a lending deposit process in the same block without contention, enabling
throughput that scales with object diversity.

[Feature 3 - DAG-Based Consensus]
Narwhal-Bullshark consensus provides deterministic ordering with minimal
latency. The DAG structure significantly reduces the surface area for MEV
extraction and front-running.

[Feature 4 - Programmable Transactions]
Multiple market actions compose into a single atomic transaction. Supply
collateral, open a derivatives position, and set a stop-loss — all in one
block, all or nothing.

========== SMART ACCOUNT ARCHITECTURE ==========

[Title]
Smart Account Architecture

[Intro]
Users interact through a self-custodial smart account that replaces seed phrases
with modern authentication while preserving full key control.

[Feature 1 - zkLogin]
Users authenticate via Google OAuth. An ephemeral keypair is generated locally,
linked to the OAuth identity through a zero-knowledge proof, and used to sign
transactions. The OAuth provider never sees the private key. The ZK proof
verifies identity without revealing it.

[Feature 2 - Multi-Signer & Social Recovery]
Accounts support multiple signers with configurable thresholds. Guardian-based
social recovery allows account restoration without seed phrases or centralized
intermediaries.

[Feature 3 - Session Permissions]
For high-frequency trading, users can delegate constrained authority to session
keys. These keys have limited scope (specific markets, max amounts) and
time-bound validity, reducing authentication friction without compromising
security.

[Feature 4 - Key Encryption]
Private keys are encrypted at rest using AES-256-GCM with PBKDF2 key derivation
(100,000 iterations). Keys are cleared from memory after use. Rate limiting
prevents brute-force attacks with progressive lockout thresholds.

[Title]
Market Infrastructure

[Intro]
All markets execute natively on-chain through Move smart contracts. No off-chain

  [Title]
  DeepBook V3 — Central Limit Order Book

  [Description]
  Fully on-chain CLOB with price-time priority matching. Supports limit orders,

  DeepBook events in real-time for the trading UI.

  Specs:
    • On-chain matching engine
    • Rust event indexer
    • Level 2 orderbook depth
    • Real-time trade streaming

[Prediction Markets Subsection]

  [Title]
  Prediction Markets

  [Description]
  Binary YES/NO outcome markets with dedicated orderbooks per outcome.
  NUSDC-collateralized. Markets resolve on-chain via admin oracle or automated
  resolution.

  Specs:
    • Binary outcome orderbooks
    • NUSDC collateral
    • On-chain resolution
    • Admin market creation

  On-Chain Lottery

  [Description]

========== CAPITAL INFRASTRUCTURE ==========

[Title]
Capital Infrastructure

[Intro]
A portfolio-level risk engine evaluates all positions under a single margin
framework. Risk is computed per account, not per protocol. This is the
infrastructure that keeps capital flowing — and keeps users from needing to
leave.

[Multi-Collateral System Subsection]

  [Title]
  Multi-Collateral System

  Items:
    • NUSDC: 100% collateral value (0% haircut)
    • NBTC: 90% collateral value (10% haircut)
    • Collateral objects transition between states: idle, margin, lending

[4-Tier Risk Thresholds Subsection]

  [Title]
  4-Tier Risk Thresholds

  Items:
    • Initial Margin (IM): 10% — Required to open new positions
    • Warning Level: 8% — Alert threshold, no new positions
    • Maintenance Margin (MM): 5% — Liquidation trigger point
    • Forced Close (FC): 3% — Full position closure

[Liquidation Mechanics Subsection]

  [Title]
  Liquidation Mechanics

  Items:
    • Partial liquidation: max 50% of position per event
    • Liquidation bonus: 5% incentive for keepers
    • Deterministic execution — no discretionary decisions
    • 10-second keeper monitoring interval

========== WHY A NEW CHAIN ==========

[Title]
Why a New Chain

[Intro]
Pado는 Sui에서 포크된 고성능 블록체인 Nasun 위에서 동작합니다. 전용 L1을
구축한 것은 이념이 아닌 아키텍처적 선택입니다.

[Feature 1 - Object-Based State Model]
모든 자산, 포지션, 마켓이 독립적인 온체인 객체로 존재하며, 각각 고유한
소유권과 접근 규칙을 가집니다. 객체는 글로벌 상태를 건드리지 않고 조합,
전송, 수정할 수 있습니다.

[Feature 2 - Parallel Execution]
독립적인 객체를 다루는 트랜잭션은 동시에 실행됩니다. 현물 거래와 렌딩
입금이 같은 블록에서 경합 없이 처리되어, 객체 다양성에 비례하여 처리량이
확장됩니다.

[Feature 3 - DAG-Based Consensus]
Narwhal-Bullshark 합의 메커니즘이 최소 지연시간으로 결정론적 순서를
제공합니다. DAG 구조는 MEV 추출과 프론트러닝의 공격 표면을 크게 줄입니다.

[Feature 4 - Programmable Transactions]
여러 마켓 액션이 하나의 원자적 트랜잭션으로 합성됩니다. 담보 제공, 파생상품
포지션 오픈, 스톱로스 설정을 한 블록에서 올오어낫싱으로 처리합니다.

========== SMART ACCOUNT ARCHITECTURE ==========

[Title]
Smart Account Architecture

[Intro]
사용자는 시드 구문을 현대적 인증으로 대체하면서도 완전한 키 제어를
유지하는 비수탁형 스마트 계정을 통해 상호작용합니다.

[Feature 1 - zkLogin]
사용자는 Google OAuth로 인증합니다. 임시 키페어가 로컬에서 생성되고,
영지식 증명을 통해 OAuth 아이덴티티에 연결되어 트랜잭션 서명에 사용됩니다.
OAuth 제공자는 개인키를 볼 수 없습니다. ZK 증명은 아이덴티티를 드러내지
않고 검증합니다.

[Feature 2 - Multi-Signer & Social Recovery]
계정은 설정 가능한 임계값으로 복수 서명자를 지원합니다. Guardian 기반
소셜 리커버리를 통해 시드 구문이나 중앙화된 중개자 없이 계정을 복구할 수
있습니다.

[Feature 3 - Session Permissions]
고빈도 트레이딩을 위해 세션 키에 제한된 권한을 위임할 수 있습니다. 세션
키는 제한된 범위(특정 마켓, 최대 금액)와 시간 제한 유효성을 가지며, 보안을
타협하지 않으면서 인증 마찰을 줄입니다.

[Feature 4 - Key Encryption]
개인키는 AES-256-GCM과 PBKDF2 키 파생(100,000 반복)을 사용하여 저장 시
암호화됩니다. 사용 후 키는 메모리에서 제거됩니다. 점진적 잠금 임계값으로
무차별 대입 공격을 방지합니다.
