# Baram - 비개발자를 위한 설명

> 작성일: 2026-02-23
> 대상: 기술 배경 없이도 Baram의 작동 원리를 이해하고 싶은 분

---

## 한줄 요약

Baram은 **AI 에이전트가 돈을 쓸 때, 그 내역을 블록체인에 자동으로 기록하고 감사할 수 있게 해주는 시스템**입니다.

---

## 일상 비유: 법인카드 관리 시스템

회사에서 직원에게 **법인카드**를 줄 때를 생각해보세요.

| 현실 세계 (법인카드) | Baram |
|----------------------|-------|
| 직원에게 법인카드 발급 | AI 에이전트에게 **Budget(예산)** 할당 |
| 일일 한도 50만원 설정 | **SpendingLimits** - 일/주/월 지출 한도 설정 |
| "식비, 교통비만 사용 가능" | **카테고리 제한** - 허용된 용도만 결제 가능 |
| 결제할 때마다 문자 알림 | **AIExecutionReport(AER)** - 매 실행마다 블록체인에 기록 |
| 월말에 사용 내역 영수증 제출 | **Dashboard** - 실시간으로 모든 실행 이력 확인 |
| 부정사용 시 카드 정지 | **Kill Switch** - 에이전트 즉시 비활성화 |

핵심 차이점: 일반 법인카드는 직원이 영수증을 "자발적으로" 제출하지만, Baram에서는 **AI가 일을 끝내면 영수증(AER)이 자동으로, 강제로 생성**됩니다. AI가 "영수증을 안 내겠다"고 할 수 없는 구조입니다.

## 작동 흐름도

```
사용자 (Agent Owner)
    │
    ├─[1] 에이전트 등록 + 예산 설정
    │       "이 AI에게 월 100 NUSDC, 일 10건까지 허용"
    │
    ▼
┌──────────────────────────────┐
│    Baram Dashboard (웹앱)     │
│    - 에이전트 관리             │
│    - 예산 설정/모니터링        │
│    - 실행 보고서 열람          │
│    - 채팅으로 AI 실행          │
└──────────┬───────────────────┘
           │
    ├─[2] 사용자가 채팅으로 AI에게 질문
    │
    ▼
┌──────────────────────────────┐
│   TEE (보안 금고 서버)        │  ← 프롬프트가 여기서 복호화/처리됨
│   AWS Nitro Enclave           │     서버 운영자도 내용을 볼 수 없음
│                               │
│   [3] AI가 답변 생성          │
│   [4] 결과 + 증명을 블록체인에│
│       제출 (자동)             │
└──────────┬───────────────────┘
           │
    ├─[5] 블록체인에 기록
    │
    ▼
┌──────────────────────────────┐
│   Nasun 블록체인 (온체인)     │
│                               │
│   - 에스크로 정산 완료         │
│   - AER(실행 보고서) 영구 저장 │
│   - 예산 차감 기록             │
└──────────┬───────────────────┘
           │
    ├─[6] Dashboard에서 결과 확인
    │
    ▼
사용자: "AI가 뭘 했는지, 얼마 썼는지 전부 보인다"
```

## 단계별 설명

### 1단계: 에이전트 등록 + 예산 설정

- **무슨 일이 일어나는지**: 사용자가 Dashboard에서 AI 에이전트를 등록하고, 그 에이전트가 쓸 수 있는 예산(Budget)을 만듭니다. 예산에는 "하루 최대 10 NUSDC", "주 50 NUSDC", "AI 추론 용도만 허용" 같은 제한을 설정합니다.
- **왜 이 단계가 필요한지**: AI에게 무제한 권한을 주면 위험합니다. 사전에 한도를 정해두면, AI가 설정을 넘어서는 행동을 할 수 없습니다. 마치 신입사원에게 사장님 법인카드가 아니라 한도가 있는 카드를 주는 것과 같습니다.

### 2단계: 채팅으로 AI에게 작업 요청

- **무슨 일이 일어나는지**: 사용자가 Dashboard의 채팅 화면에서 AI에게 질문이나 작업을 요청합니다. 이때 NUSDC(스테이블코인)가 에스크로(Escrow, 임시 보관)로 잠깁니다.
- **왜 이 단계가 필요한지**: 에스크로는 "일 끝나면 돈 줄게"를 보장하는 장치입니다. AI 서비스 제공자(Executor)는 "일했는데 돈을 안 받을까봐" 걱정할 필요가 없고, 사용자는 "돈만 가져가고 일 안 하면 어쩌지" 걱정할 필요가 없습니다.

### 3단계: TEE에서 AI가 답변 생성

- **무슨 일이 일어나는지**: 사용자의 질문은 암호화되어 TEE(Trusted Execution Environment)라는 특수한 보안 공간으로 전달됩니다. TEE 안에서만 질문이 복호화되고, AI가 답변을 생성합니다.
- **왜 이 단계가 필요한지**: TEE는 "밀봉된 유리 금고" 같은 것입니다. 안에서 무슨 일이 일어나는지 바깥에서 볼 수 있지만(투명성), 안의 내용물을 꺼내거나 바꿀 수는 없습니다(변조 방지). 서버를 운영하는 사람조차도 사용자의 프롬프트를 엿볼 수 없습니다.

### 4단계: 결과 + 증명을 블록체인에 제출

- **무슨 일이 일어나는지**: AI가 답변을 생성하면, Executor(AI 서비스 운영자)가 그 결과를 블록체인에 제출합니다. 이때 한 번의 트랜잭션으로 4가지 일이 동시에 일어납니다:
  1. 에스크로 정산 (Executor에게 대가 지급)
  2. 정산 영수증(SettlementReceipt) 발행
  3. AER(실행 보고서) 생성 - 영수증을 "소비"해야만 생성 가능
  4. Executor 실적 업데이트
- **왜 이 단계가 필요한지**: 4가지가 "하나의 묶음(원자적 트랜잭션)"으로 처리되기 때문에, "돈은 받았는데 보고서는 안 남기겠다"가 물리적으로 불가능합니다. 이것이 Baram의 핵심 설계입니다.

### 5단계: 블록체인에 영구 기록

- **무슨 일이 일어나는지**: AER(AIExecutionReport)이 NFT 형태로 사용자의 지갑에 저장됩니다. 이 보고서에는 8개 카테고리, 31개 필드의 상세 정보가 들어있습니다.
- **왜 이 단계가 필요한지**: 블록체인에 기록되면 누구도 수정하거나 삭제할 수 없습니다. 나중에 "AI가 이상한 일을 했다"는 의심이 있을 때, 이 기록이 객관적인 증거가 됩니다.

### 6단계: Dashboard에서 결과 확인

- **무슨 일이 일어나는지**: 사용자가 Dashboard에서 AI의 답변을 보고, Execution Reports 탭에서 온체인에 기록된 실행 보고서를 확인합니다. 예산 사용량, 남은 한도도 실시간으로 볼 수 있습니다.
- **왜 이 단계가 필요한지**: 관리자가 모든 활동을 한눈에 파악할 수 있어야 합니다. "이 에이전트가 이번 주에 얼마나 썼지?", "어떤 종류의 작업을 했지?"를 바로 확인할 수 있습니다.

## 왜 이렇게 만들었는가 - 설계 결정

### 왜 블록체인인가? (vs 일반 데이터베이스)

| | 일반 DB | 블록체인 |
|---|---|---|
| 기록 수정 | 관리자가 수정/삭제 가능 | 아무도 수정 불가 |
| 신뢰 | "우리를 믿어주세요" | "코드가 보장합니다" |
| 감사 | 내부 감사만 가능 | 누구나 검증 가능 |

AI가 돈을 다루는 상황에서 "기록을 못 바꾼다"는 것이 핵심 가치입니다. 사용자가 서비스 운영자를 신뢰하지 않아도 되는 구조를 만든 것입니다.

### 왜 TEE인가? (vs 일반 서버)

일반 서버에서 AI를 돌리면, 서버 운영자가 사용자의 프롬프트(질문 내용)를 볼 수 있습니다. TEE(AWS Nitro Enclave)를 사용하면:

- 서버 운영자도 프롬프트를 볼 수 없음
- AI 모델이 변조되지 않았음을 하드웨어가 증명 (PCR Attestation)
- "어떤 코드가 어떤 환경에서 실행됐는지"를 암호학적으로 검증 가능

단점: 일반 서버보다 비용이 높고, 사용할 수 있는 AI 모델에 제약이 있습니다 (현재 LLaMA 3.2 3B 모델 사용).

### 왜 영수증을 "강제"하는가? (Hot-Potato 패턴)

Baram에서 가장 독특한 설계는 **SettlementReceipt**(정산 영수증) 패턴입니다.

Move 언어에는 `drop` 능력(ability)이라는 개념이 있습니다. `drop` 능력이 없는 객체는 "버릴 수 없는 뜨거운 감자(hot potato)"처럼 반드시 어딘가에 사용(소비)해야 합니다.

```
Executor가 정산 완료
    ↓
SettlementReceipt 발행 (drop 능력 없음!)
    ↓
이 영수증을 "소비"할 수 있는 유일한 방법 = AER 생성
    ↓
AER이 자동으로 생성됨 (선택이 아닌 필수)
```

이 설계 덕분에 "정산은 했지만 보고서는 안 남기겠다"가 코드 레벨에서 불가능합니다. 스마트 계약이 물리적으로 이를 허용하지 않습니다.

## 시스템 구성 요소 요약

| 구성 요소 | 비유 | 역할 | 상태 |
|-----------|------|------|------|
| **Dashboard** | 관리자 콘솔 | 에이전트 등록, 예산 설정, 보고서 열람, AI 채팅 | 작동 중 |
| **Budget** | 법인카드 한도 | 일/주/월 지출 한도, 카테고리 제한, 속도 제한 | 작동 중 |
| **Escrow** | 에스크로 계좌 | 결제 금액을 임시 보관, 작업 완료 시 자동 지급 | 작동 중 |
| **AER** | 블랙박스 기록 | 8카테고리 31필드의 실행 보고서 (온체인 NFT) | 작동 중 |
| **TEE** | 밀봉된 금고 | 프롬프트 프라이버시 보호 + 실행 무결성 증명 | 작동 중 |
| **Executor** | AI 서비스 운영자 | TEE에서 AI를 돌리는 사업자 (스테이킹으로 담보) | 작동 중 |
| **Kill Switch** | 비상 정지 버튼 | 에이전트를 즉시 비활성화하여 추가 지출 차단 | 작동 중 |
| **Agent Runner** | 자율 직원 | 로컬에서 실행, 주기적으로 AI 호출 + Budget 차감 + AER 기록 | 미구현 |

## AER(AIExecutionReport)의 8가지 카테고리

하나의 AER에는 AI 실행에 대한 **8가지 관점**의 정보가 기록됩니다:

| # | 카테고리 | 기록하는 내용 | 비유 |
|---|----------|---------------|------|
| 1 | **Identity** | 누가 요청했고, 어떤 에이전트가 실행했는지 | 주문서의 "주문자/배달원" 정보 |
| 2 | **Execution** | 어떤 AI 모델을 사용했고, 얼마나 걸렸는지 | 작업 일지의 "사용 장비/소요 시간" |
| 3 | **Financial** | 얼마를 청구했고, 얼마가 Executor에게 갔는지 | 영수증의 "금액/수수료" 항목 |
| 4 | **Content** | 프롬프트/응답의 해시값 (내용 자체는 비공개) | 봉인된 편지의 "봉인 도장" |
| 5 | **Quality** | 토큰 수, 지연 시간 등 품질 지표 | 배달 앱의 "배달 시간/평점" |
| 6 | **Security** | TEE 인증 상태, PCR 해시값 | 식품의 "위생 검사 인증서" |
| 7 | **Delegation** | 에이전트가 위임받아 실행했는지 여부 | "대리인이 서명한 계약서" 표시 |
| 8 | **Chaining** | 이전 실행과 연결된 작업인지 여부 | "이전 주문 연장" 표시 |

## 현재 작동 중인 기능 vs 향후 비전

| 기능 | 현재 상태 | 비유 |
|------|-----------|------|
| Agent 등록 + Budget 설정 | **작동 중** | 직원에게 법인카드 발급 완료 |
| 채팅을 통한 AI 실행 + AER 기록 | **작동 중** | 사장이 직접 법인카드를 긁고 영수증을 받는 것 |
| TEE (프라이버시 보호 실행) | **작동 중** | 밀봉된 금고에서 작업 처리 |
| Kill Switch + 예산 한도 | **작동 중** | 카드 정지 + 일일 한도 |
| **Agent 자율 실행** | **미구현** | 직원이 알아서 결제하고 영수증이 자동으로 쌓이는 것 |
| **BYO Agent (자체 LLM 사용)** | **미구현** | 직원이 자기가 찾은 거래처에서 구매하고 영수증만 제출 |
| 다중 Executor 경쟁 | **미구현** | 여러 배달 회사 중 가장 좋은 곳을 선택 |

현재 Baram은 "사장이 직접 법인카드를 긁고 영수증을 받는" 단계입니다. 핵심 인프라(예산, 한도, AER 강제 기록, Kill Switch)는 모두 온체인에서 작동합니다. 다음 단계는 "직원(AI Agent)에게 한도 카드를 줘서 알아서 결제하게" 만드는 것입니다.

## 현재 한계점 및 주의사항

| 한계 | 설명 | 향후 계획 |
|------|------|-----------|
| **Devnet 전용** | 현재 Nasun Devnet에서만 작동하며, 실제 자산 거래는 불가 | Mainnet 출시 시 실제 자산 지원 |
| **TEE 모델 제한** | TEE 내에서는 LLaMA 3.2 3B 소형 모델만 실행 가능 | 더 큰 모델은 Lambda(Groq)로 우회 중 |
| **단일 Executor** | 현재 테스트용 Executor 1개만 운영 중 | 다중 Executor 경쟁 체계 구축 예정 |
| **베타 접근** | BetaAccessNFT 소유자만 채팅 사용 가능 | 정식 출시 시 개방 예정 |
| **프롬프트 비공개** | AER에 프롬프트 해시만 기록, 원문은 사용자 브라우저에만 저장 | 선택적 공개 기능 검토 중 |
| **Agent 자율 실행 미구현** | Agent가 스스로 판단/실행/기록하는 흐름이 없음 | Agent Runner + Lambda 연동 구현 예정 |
