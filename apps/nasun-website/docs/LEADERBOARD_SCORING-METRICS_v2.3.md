## 점수 계산 공식 완전 가이드

리더보드 점수는 크게 2단계에 걸쳐 계산됩니다. 먼저 장기적인 활동과 영향력을 나타내는 `totalScore`를 계산한 후, 여기에 단기적인 활동 추세를 반영하여 최종 순위의 기준이 되는 `finalScore`를 계산합니다.

### Total Score vs Final Score: 최종 점수 계산의 두 단계

| 구분 | **`totalScore` (누적 기본 점수)** | **`finalScore` (최종 순위 결정 점수)** |
| :--- | :--- | :--- |
| **목적** | 사용자의 장기적인 활동량과 영향력을 반영하는 **'기본 점수'** | 단기적인 활동 추세(보너스/패널티)를 반영하여 순위를 결정하는 **'최종 조정 점수'** |
| **계산 시점** | 파이프라인 8단계: `ScoreCalculatorTask` | 파이프라인 10단계: `LeaderboardGeneratorTask` |
| **역할** | `finalScore`를 계산하기 위한 기초 값 | 프론트엔드 리더보드 순위를 정하는 실제 기준 |
| **주요 변수** | 인게이지먼트(좋아요 등), 팔로워 수, 언어 | `totalScore`, 최근 60일/7일 활동, 비활동 기간 |

---

### 1단계: `totalScore` (누적 기본 점수) 계산

`totalScore`는 사용자의 순수 인게이지먼트 활동과 영향력을 기반으로 계산되는 **누적 점수**입니다.

**계산 공식:**
> `totalScore` = (A: 기본 인게이지먼트 점수 합계) × (B: 커뮤니티 가중치)

#### A. 기본 인게이지먼트 점수 체계

사용자가 얻은 인게이지먼트 유형별로 아래의 기본 점수를 합산합니다.

| 항목       | 점수 | 관리 방식                        |
| ---------- | ---- | -------------------------------- |
| `likes`    | 0.2  | 환경변수 `SCORE_WEIGHT_LIKES`    |
| `replies`  | 0.4  | 환경변수 `SCORE_WEIGHT_REPLIES`  |
| `reposts`  | 0.4  | 환경변수 `SCORE_WEIGHT_REPOSTS`  |
| `quotes`   | 0.6  | 환경변수 `SCORE_WEIGHT_QUOTES`   |
| `mentions` | 0.5  | 환경변수 `SCORE_WEIGHT_MENTIONS` |

**참고**: 과거에 사용되었던 멘션/인용에 대한 복잡한 고급 스코어링(품질, 쿨다운 보너스) 로직은 현재 코드에서 사용되지 않으며, 모든 인게이지먼트는 환경변수에 설정된 동일한 기본 점수를 받습니다.

#### B. 커뮤니티 가중치 시스템 (Community Weight System)

기본 인게이지먼트 점수 합계에 곱해지는 '배수'입니다. 사용자의 **영향력(팔로워 수)**과 **언어(한국어/글로벌)**를 기반으로 계산됩니다.

**가중치 계산 공식:**

```typescript
// 1. 팔로워 수 기반 가중치 계산
const followerWeight = Math.min(Math.log(followers + 1) / Math.log(config.logBase), config.maxCap);

// 2. 언어 배수 적용
const finalWeight = followerWeight * config.languageMultiplier;
```

**가중치 설정값:**

*이 설정값들은 `.env` 파일 및 코드에 정확히 구현되어 있습니다.*

```typescript
// 한국 커뮤니티 (높은 영향력 보상)
KOREAN_CONFIG = {
  logBase: 8,        // 낮은 로그 베이스 (팔로워 수 증가에 따른 가중치 상승이 빠름)
  multiplier: 1.02,  // 2% 추가 가산
  maxCap: 5.0       // 팔로워 가중치의 최대 상한
}

// 글로벌 커뮤니티 (기본 영향력 보상)  
GLOBAL_CONFIG = {
  logBase: 30,       // 높은 로그 베이스 (가중치 상승이 완만함)
  multiplier: 1.0,   // 기본값
  maxCap: 4.0       // 팔로워 가중치의 최대 상한
}
```

### 2단계: `finalScore` (최종 순위 결정 점수) 계산

`finalScore`는 `totalScore`에 단기적인 활동 추세를 반영하여 계산되는 **최종 순위 결정 점수**입니다.

**계산 공식:**
> `finalScore` = `totalScore` + ① `activeDaysScore` + ② `activityBonus` + ③ `inactivityPenalty`

#### ① `activeDaysScore` (동점자 처리 점수)
*   **목적:** `finalScore`가 동일할 경우 순위를 결정하기 위한 보조 점수입니다.
*   **계산:** 최근 60일간의 활동 일수 × 0.1점 (최대 6.0점)

#### ② `activityBonus` (활동 보너스)
*   **목적:** 꾸준한 활동을 보상합니다.
*   **계산:** 최근 7일 중 3일 이상 활동 시, 활동일에 따라 **최대 +1.4점**의 보너스를 받습니다. `(활동일수 - 3 + 1) * 0.28`

#### ③ `inactivityPenalty` (비활동 패널티)
*   **목적:** 장기 비활동에 대한 불이익을 줍니다.
*   **계산:** 3일 이상 활동이 없을 경우, 비활동 기간에 따라 **최대 -5.0점**까지 감점됩니다. `(비활동일수 - 2) * -0.3`

### 📅 수집 방식: 일일 스냅샷 (Daily Snapshot)

현재 시스템은 매일 실행될 때마다 X API가 허용하는 기간 내의 모든 관련 활동을 **하나의 완전한 스냅샷**으로 가져옵니다. 

- **수집 대상**:
    - **Passive 지표 (좋아요, 리트윗, 인용)**: **3일 전** 트윗 대상
    - **Active 지표 (멘션, 답글)**: **1일 전** 트윗 대상
- **데이터 처리**: 매번 새로운 스냅샷을 기반으로 점수를 계산하며, 점수 중복 계산 방지는 아래 설명된 **멱등성 필터**를 통해 100% 보장됩니다.

### 🔐 멱등성 보장 메커니즘 (점수 중복 계산 방지)

파이프라인이 하루에 여러 번 실행되더라도 동일한 활동에 대한 점수가 중복으로 누적되지 않도록, **`delta-calculator.ts`의 `calculateSnapshotScores` 함수 내에 "멱등성 필터"**가 구현되어 있습니다.

**동작 원리**:

1.  **활동 기록 조회**: 점수 계산기는 계산을 시작하기 전에, `RECENT#` 레코드를 조회하여 **오늘 이미 점수 계산이 완료된 활동들의 목록**을 가져옵니다.
2.  **신규 활동 필터링**: 새로 수집된 스냅샷 데이터와 '오늘 이미 처리된 활동 목록'을 비교하여, 스냅샷에만 존재하는 **'순수 신규' 활동만 필터링**합니다.
3.  **점수 계산**: 오직 이 '순수 신규' 활동들에 대해서만 점수를 계산합니다.
4.  **활동 기록 저장**: 점수 계산이 완료되면, 방금 처리한 활동들을 `RECENT#` 레코드로 데이터베이스에 저장하여 다음 실행 시 중복 처리되지 않도록 합니다.

**결론**: 이 멱등성 필터 덕분에, 파이프라인이 몇 번을 재실행하더라도 **동일한 활동은 단 한 번만 점수화**되는 것이 완벽하게 보장됩니다.
