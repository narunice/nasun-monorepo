## 📋 X API Basic Plan Rate Limits

### 일일 제한 (App-only auth)
- **총 요청 수**: 1,667 requests/day
- **사용률 목표**: 50% 이하 (833 requests/day)
- **안전 여유**: 25% (417 requests/day 여유분)

### 15분 제한 (User context)
- **Tweet lookup**: 15 requests/15분
- **User timeline**: 15 requests/15분  
- **Users lookup**: 15 requests/15분
- **안전 목표**: 10 requests/15분 (66% 사용률)

## 🔍 수집 타입별 API 사용량 분석

### 1. V1 인게이지먼트 수집 (Daily Data Collection)

#### 사용 엔드포인트
```
/2/users/by/username/{username}     - 1회/일 (타겟 사용자 조회)
/2/users/{id}/tweets               - 1회/일 (타겟 트윗 목록)
/2/tweets/{id}/liking_users        - N회/일 (트윗별 좋아요 사용자)
/2/tweets/{id}/retweeted_by        - N회/일 (트윗별 리트윗 사용자)
```

#### 예상 호출량
- 기본 호출: 2회/일
- 트윗당 인게이지먼트: 2-3회 × 평균 10개 트윗 = 25회/일
- **총합**: ~46회/일
- **15분 분산**: 3-4회/15분

### 2. V2 누적 데이터 수집 (Cumulative Data Collector)

#### 사용 엔드포인트
```
/2/users/by/username/{username}     - 1회/일
/2/users/{id}/tweets               - 1회/일
/2/tweets/{id}/liking_users        - N회/일
/2/tweets/{id}/retweeted_by        - N회/일
/2/users/{id}/bookmarks            - N회/일 (OAuth 2.0 필요)
```

#### 예상 호출량
- 기본 호출: 2회/일
- 인게이지먼트 수집: 30회/일 (V1보다 약간 많음)
- 북마크 수집: 5-10회/일 (OAuth 활성화 시)
- **총합**: ~50회/일
- **15분 분산**: 3-4회/15분

### 3. 리트윗 보너스 수집 (Phase 1 추가)

#### 사용 엔드포인트
```
/2/users/{id}/tweets               - 1회/일 (타겟 리트윗 타임라인)
/2/tweets/{id}                     - N회/일 (원본 트윗 조회)
```

#### 예상 호출량
- 타임라인 호출: 1회/일
- 원본 트윗 조회: 최대 100회/일 (설정값)
- **총합**: 101회/일 (최대치)
- **⚠️ 위험**: 15분 내 100회 호출 시 Rate Limit 초과!

## 🚨 Rate Limit 위험 분석

### High Risk: 리트윗 보너스 수집
```
문제: 15분 내 최대 100회 API 호출
Rate Limit: 15회/15분
위험도: CRITICAL (666% 초과)
```

### Medium Risk: 동시 실행
```
V1 + V2 + 리트윗보너스 동시 실행 시:
15분 내 최대: 4 + 4 + 100 = 108회
Rate Limit: 15회/15분  
위험도: HIGH (720% 초과)
```

### Low Risk: 개별 시스템
```
V1 단독: 3-4회/15분 (26% 사용률) ✅
V2 단독: 3-4회/15분 (26% 사용률) ✅
```

## 💡 최적화 전략

### 1. 리트윗 보너스 수집 최적화

#### 방법 1: 배치 처리 (권장)
```
현재: 1회에 100개 처리
개선: 10개씩 10배치로 분할
시간 간격: 15분마다 1배치
총 소요시간: 150분 (2.5시간)
안전성: HIGH ✅
```

#### 방법 2: 시간 분산
```
현재: 13:30 UTC 1회 실행
개선: 13:30, 14:00, 14:30, 15:00 (4회 분할)
배치당 호출: 25회 (Rate Limit 초과!)
안전성: LOW ❌
```

#### 방법 3: 혼합 접근 (최종 권장)
```
배치 크기: 8개/배치
배치 간격: 15분
총 배치: 13개 (100개÷8)
총 소요시간: 195분 (3.25시간)
15분 사용률: 53% (8/15) ✅
안전 여유: 47%
```

### 2. 시간대 분산 전략

#### 현재 (위험)
```
00:00 UTC: V1 시스템 (4회/15분)
01:00 UTC: V2 시스템 (4회/15분)  
04:30 UTC: 리트윗 보너스 (100회/15분!) ❌
```

#### 개선안 (안전)
```
00:00 UTC: V1 시스템 (4회/15분)
02:00 UTC: V2 시스템 (4회/15분)
06:00 UTC: 리트윗 보너스 시작 (8회/15분)
06:15 UTC: 배치 2 (8회/15분)
...
09:00 UTC: 배치 13 완료
```

### 3. 적응형 백오프 전략

#### Rate Limit 감지 시
```
429 응답 수신 → 15분 대기
재시도 → 성공 시 계속
재실패 → 30분 대기 (지수 백오프)
최대 대기: 4시간
```

#### 예방적 제한
```
15분 내 호출 수 추적
8회 도달 시 → 자동 대기
여유 호출 수 모니터링
```
